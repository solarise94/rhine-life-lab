# OAA-2 Workboard Fuel Buffer And Auto Episode Completion

Status: design note (approved for implementation).

Date: 2026-06-02

Last refined: 2026-06-02 (tightened into implementation order + data-structure changes + consumption semantics + transient payload contract)

Supersedes: earlier terminal-summary wake-queue idea. This revision is
state-driven, not queue-driven. The rectifier and wake-signal queue are
removed; wake signals are derived from state.

## 1. Contract Summary

- `/auto <objective>` starts a single scoped auto episode.
- The workboard answers one question: "is there wake fuel for Manager?".
- Manager wakes to evaluate fuel, not because every terminal event is
  actionable.
- `complete` is a distinct confirmation phase, not an auto-stop.
- The episode ends only when Manager calls `finish_auto_episode` (or a
  stop/risk-control path ends it).

Non-goals: no change to owner-session locking; no non-owner takeover; no
always-on auto restoration; no workboard-level start validation.

## 2. Wake Fuel Kinds

| Kind | Source | Persistent | Exit rule |
| --- | --- | --- | --- |
| `todo` | Manager adds a `planned` card | yes | `skip`; or auto-exits once the card is submitted/started (it is no longer `planned`, so it is no longer fuel) |
| `complete_signal` | a card first enters `complete` | no | consumed only after Manager explicitly acknowledges, skips, or converts the signal into a new card action; invalidated by later card change; entering an evaluate turn marks the signal as `seen_in_revision` but does NOT remove it |
| `block_signal` | a card first enters `failed` / blocked terminal | no | same consumption rule as `complete_signal` — explicit Manager action required; not consumed by merely emitting `workboard_evaluate` |

Constraints:

- `todo` only accepts cards that are currently `planned`.
- `todo` cannot be marked `done`; the only explicit fuel-exit is `skip`.
  Running the card is not a fuel operation — submit/start moves the card
  out of `planned`, and fuel membership is recomputed from state.
- `complete_signal` and `block_signal` are edge-triggered, not mirrors.
- Consumption of `complete_signal` / `block_signal` requires explicit
  Manager acknowledgement/skip/conversion, not mere evaluation-turn entry.
  Entering the evaluate turn only marks the signal `seen_in_revision` so the
  same fuel is visible to the next evaluate turn if Manager does not act on
  it. Crashing or timing out during evaluation does not lose fuel.
- The workboard does not answer "can this card really start?".

## 3. Start Validation Boundary

Start validation is owned by the normal `ready_to_do` /
`workboard_card_state` / start pipeline. The workboard lane only reports
which planned cards Manager has asked to run.

- a `todo` item may exist while dependencies are unsatisfied;
- submit/start re-validates through the existing pipeline;
- submission failure surfaces through the normal card/block path, not
  through workboard semantics.

## 4. Episode State Machine

| State | Entry condition | Notes |
| --- | --- | --- |
| `running` | any Manager turn, tool call, run, or job in flight | |
| `idle` | `has_running=true` and `has_wake_fuel=false` | |
| `pending_wake` | `has_running=false` and `has_wake_fuel=true` | name is authoritative (replaces earlier "job break" wording) |
| `complete` | `has_running=false`, `has_wake_fuel=false`, not finished | evaluation phase, not terminal |
| `finished` | finish tool or stop/risk-control path | true terminal |

Stop reasons (`user_stop`, `wake_storm`, `chain_limit`, internal error) are
observable as stop metadata, not states.

## 5. Wake Types

| Wake kind | Trigger | Purpose |
| --- | --- | --- |
| `workboard_evaluate` | state enters `pending_wake` | Manager inspects fuel and submits/skips/adjusts |
| `complete_evaluate` | state enters `complete` | Manager confirms completion or continues |

Wake signals are derived from current state + a one-shot latch. There is no
wake queue, so queue-level dedup is unnecessary. The latch only prevents
re-emitting the same wake while state is unchanged (Section 7).

**These are NOT persistent queue events.** The payloads in Section 8.6 are
transient notification snapshots generated by each evaluate-loop iteration.
They are not stored on disk, not subject to queue ack/dequeue semantics, and
not reified as standalone objects. The only persistent fields are the latch
fields on `AutoEpisode` (Section 7). `wake_in_flight` is a runtime
boolean held by the `AutoEpisode` record: set `true` when Manager begins an
evaluate turn, cleared back to `false` when the turn settles. If no
`AutoEpisode` is active, no evaluate loop runs and no transient payload is
generated.

## 6. Finish Tool Contract

Tool: `finish_auto_episode`.

- valid only when auto state is `complete`;
- transitions the episode to `finished`;
- returns the final Complete message directly — no follow-up wake;
- the Complete prompt asks Manager to summarize objective, completion
  status, outputs, and remaining caveats.

If Manager instead creates new fuel during the `complete` turn (new `todo`,
card adjustment that emits `complete_signal` / `block_signal`), state
leaves `complete` and re-enters `pending_wake` or `running` — no finish
tool call occurs.

## 7. Latch And Risk Controls

### State latch (not a queue)

Minimum fields on the auto-episode record:

| Field | Purpose |
| --- | --- |
| `fuel_revision` | increments whenever wake fuel changes |
| `wake_in_flight` | true while a Manager evaluate turn is active; set `true` when the evaluate turn begins, cleared to `false` when the turn settles (success, error, or timeout). This is a runtime mutex, not a persistent queue gate — it prevents concurrent evaluate turns within one episode |
| `last_notified_revision` | fuel revision already notified |
| `completion_notified` | one-shot latch for `complete_evaluate` |

Emit rules:

```text
if state == pending_wake
   and not wake_in_flight
   and fuel_revision > last_notified_revision:
       emit workboard_evaluate

if state == complete
   and not completion_notified:
       emit complete_evaluate
       set completion_notified = true
```

Any fuel change clears `completion_notified` so re-entry into `complete`
re-emits exactly once.

### Risk controls

| Guard | Action |
| --- | --- |
| >5 wakes within 1 minute | stop episode, reason `wake_storm` |
| `chain_count > 50` | stop episode, reason `chain_limit` |

`chain_count` increments once per completed Manager evaluation turn, not
per state recomputation.

## 8. Data Structure Changes

### 8.1 `AutoEpisode` (persisted per project)

Add / replace:

```text
state: "running" | "idle" | "pending_wake" | "complete" | "finished"
objective: str
owner_session_id: str
fuel_revision: int
last_notified_revision: int
wake_in_flight: bool
completion_notified: bool
chain_count: int
wake_window: list[timestamp]   # rolling window for wake_storm guard
stop_reason: str | None        # user_stop | wake_storm | chain_limit | error | None
started_at: timestamp
finished_at: timestamp | None
```

Remove: any `wake_queue`, `pending_summary_wake`, or rectifier-related
fields inherited from earlier drafts.

### 8.2 `WorkboardItem` (fuel view)

Add:

```text
fuel_kind: "todo" | "complete_signal" | "block_signal"
fuel_added_at: timestamp
fuel_seen_at_revision: int | None   # fuel_revision at which Manager evaluation turn first observed this item
fuel_consumed_at: timestamp | None  # set only after Manager explicitly
                                    # acknowledges / skips / converts this
                                    # fuel into a new card action. Never
                                    # set merely because workboard_evaluate
                                    # was emitted.
```

`todo` items additionally store `card_id` and are auto-removed from the
fuel view when the referenced card leaves `planned`.

`complete_signal` / `block_signal` items store `card_id` and
`triggered_by_status`. Deduplication is on `(card_id, triggered_by_status)`
— a card that re-enters the same terminal state (e.g. re-failed) does not
generate a second signal, which is the intended semantics.

**Consumption lifecycle for `complete_signal` / `block_signal`:**

Signals are **persisted to `state.items`** immediately when a card first enters
a triggering terminal status. They are never derived at view time; this ensures
`fuel_seen_at_revision` and `fuel_consumed_at` are durable across reads.

0. Signal persisted to `state.items` when the card first enters the triggering
   status → `fuel_added_at` set, both `fuel_seen_at_revision` and
   `fuel_consumed_at` are `None`.
1. Evaluate turn begins → signal's `fuel_seen_at_revision` set to current
   `fuel_revision`. Signal stays in the fuel view.
2. Manager explicitly acts on this signal (acknowledge, skip, or convert to
   a new card action) → `fuel_consumed_at` set, signal removed from
   active-fuel view. `get_background_workboard()` will no longer surface it.
3. If Manager does NOT act on this signal during the turn (crash, timeout,
   "just looked"), the signal remains visible in the next evaluate turn
   because `fuel_consumed_at` is still `None`. `fuel_seen_at_revision` may
   be updated again on the next turn.
4. If the card changes state later (e.g. re-run), the signal is invalidated
   (`fuel_consumed_at` is set) and removed from the active-fuel view. The
   persisted item remains in `state.items` as a permanent dedup marker so the
   same `(card_id, triggered_by_status)` pair never re-emits.

Signal IDs are status-aware: `fuel_signal:{card_id}:accepted`,
`fuel_signal:{card_id}:rejected`, and `fuel_signal:{card_id}:failed`. This
makes `accepted` and `rejected` independent dedup markers; cycling through
terminal states does not overwrite a previous marker.

### 8.3 `WorkboardFuelSnapshot` (derived, not persisted)

Returned by `get_wake_fuel(project_id)`:

```text
todo_count: int
complete_signal_count: int
block_signal_count: int
top_item_ids: list[str]
top_card_ids: list[str]
fuel_revision: int
```

Used to populate `workboard_evaluate` and `complete_evaluate` payloads.

### 8.4 `finish_auto_episode` Manager tool

Input: none beyond the standard project/session context.

Output on success:

```text
{
  "ok": true,
  "state": "finished",
  "complete_message": "<manager-authored summary>",
  "stopped_at": timestamp,
}
```

Output on rejection:

```text
{
  "ok": false,
  "error_code": "auto_not_complete",
  "current_state": "<state>",
}
```

### 8.5 `workboard_card_state` response

No schema change. Behavior clarification: the response continues to drive
`ready_to_do` / start validation; this revision only adds that the
workboard fuel view reads `workboard_card_state` to decide whether a
`todo` item should auto-exit (card no longer `planned` ⇒ fuel exits).

### 8.6 Wake event payloads

These payloads are **transient notification snapshots** generated each time
the evaluate loop decides to wake Manager. They are not persisted on disk,
not stored in a queue, and not reified as standalone objects. When the
Manager wake prompt is constructed, the evaluate loop reads current
`AutoEpisode` latch fields + `WorkboardFuelSnapshot` and builds the payload
inline. No payload survives beyond the wake-prompt composition step.

`workboard_evaluate`:

```text
{
  "kind": "workboard_evaluate",
  "fuel_counts": { "todo": int, "complete_signal": int, "block_signal": int },
  "top_item_ids": list[str],
  "top_card_ids": list[str],
  "fuel_revision": int,
}
```

`complete_evaluate`:

```text
{
  "kind": "complete_evaluate",
  "prompt": "当前任务似乎已完成，请回顾任务总结情况。如果确认完成，请调用 finish_auto_episode；如果仍需继续，请调整 card 或加入新的 todo。",
  "fuel_revision": int,
}
```

## 9. Implementation Order

Phase 1 — fuel buffer separation (backend):

1. add `AutoEpisode` fields (Section 8.1) and drop legacy `wake_queue` /
   rectifier fields;
2. implement `WorkboardItem.fuel_kind` write paths (Section 8.2):
   - `todo` creation gated on card state `planned`;
   - `todo` auto-exit on submit/start (card leaves `planned`);
   - `todo` explicit `skip`;
   - reject `done` on `todo` items;
   - edge-triggered `complete_signal` / `block_signal` deduplicated on
     `(card_id, triggered_by_status)`;
3. implement `get_wake_fuel` snapshot (Section 8.3);
4. rewire state derivation: `state = f(has_running, has_wake_fuel,
   episode.finished)`.

Phase 2 — latch + wake emission:

5. implement `fuel_revision`, `wake_in_flight`,
   `last_notified_revision`, `completion_notified` latch (Section 7);
6. replace existing wake paths with state-derived emission:
   - entering `pending_wake` ⇒ `workboard_evaluate`;
   - entering `complete` ⇒ `complete_evaluate`;
7. wire `wake_storm` and `chain_limit` guards; `chain_count` increments
   per completed Manager evaluation turn.

Phase 3 — finish tool:

8. add `finish_auto_episode` tool to the Manager tool registry
   (Section 8.4);
9. gate the tool on `state == complete`; transition to `finished` and
   return the Complete message inline;
10. ensure Manager creating new fuel during a `complete` turn exits
    `complete` back to `pending_wake` / `running` without finish.

Phase 4 — start-validation decoupling:

11. confirm `ready_to_do` / `workboard_card_state` / start pipeline owns
    all start checks;
12. remove any workboard-side "can start?" predicates that leaked into
    fuel evaluation;
13. ensure `todo` presence does not override a real start failure — the
    failure still surfaces as a card/block path.

Phase 5 — manager-agent prompt + tool wiring:

14. document the two-wake contract in the Manager system prompt;
15. document `todo` rules: only `planned`, only `skip`, auto-exit on
    submit/start;
16. document the `complete` turn: either call `finish_auto_episode` or
    create new fuel.

Phase 6 — regression tests (Section 10).

## 10. Regression Tests

Backend unit:

- `todo` rejects non-`planned` cards;
- `todo` rejects `done` transition; only `skip` is accepted;
- submit/start of a `todo`-referenced card auto-exits it from fuel;
- `complete_signal` / `block_signal` do not duplicate for the same
  `(card_id, triggered_by_status)`;
- `complete_signal` / `block_signal` survive an evaluate turn where
  Manager does not explicitly act on them (`fuel_consumed_at` stays
  `None`, signal remains in active-fuel view for the next turn);
- `complete_signal` / `block_signal` are consumed (removed from
  active-fuel view) only after Manager explicitly
  acknowledges/skips/converts them; mere `workboard_evaluate` emission
  does not consume;
- `fuel_revision` increments on every fuel mutation;
- `workboard_evaluate` emits only when `fuel_revision >
  last_notified_revision` and `wake_in_flight=false`;
- `complete_evaluate` emits exactly once per `completion_notified=false`
  entry;
- fuel change during `complete` clears `completion_notified`;
- finish tool is rejected when `state != complete`;
- finish tool transitions to `finished` and returns the Complete message
  inline without another wake;
- `wake_storm` stops the episode after >5 wakes in 1 minute;
- `chain_limit` stops the episode after >50 completed Manager evaluation
  turns;
- owner-session locking is unchanged (only owner processes wakes and
  mutates the episode).

Backend integration:

- full cycle: `/auto <objective>` ⇒ cards run ⇒ `complete` ⇒
  `complete_evaluate` ⇒ `finish_auto_episode` ⇒ `finished`;
- cycle with Manager continuing past `complete`: fuel creation returns
  the episode to `pending_wake` / `running`;
- start validation failure on a `todo` card surfaces through the normal
  block path and does not suppress fuel accounting.

Manager-agent:

- Manager does not call `finish_auto_episode` outside `complete`;
- Manager summarizes objective/completion/outputs/caveats in the
  Complete message.

## 11. Acceptance Criteria

1. `/auto <objective>` is a single scoped episode.
2. Workboard answers only "is there wake fuel?".
3. Fuel kinds are exactly `todo`, `complete_signal`, `block_signal`.
4. `todo` is `planned`-only, `skip`-only, and auto-exits on
   submit/start.
5. Start validation is owned by `ready_to_do` / `workboard_card_state`,
   not by the workboard.
6. `complete` emits one `complete_evaluate`; Manager either finishes or
   continues.
7. `finish_auto_episode` is valid only in `complete`, transitions to
   `finished`, and returns the Complete message inline.
8. Repeated unchanged `pending_wake` / `complete` states do not re-emit
   wakes.
9. `wake_storm` and `chain_limit` guards stop pathological loops.
10. Owner-session locking is unchanged.
