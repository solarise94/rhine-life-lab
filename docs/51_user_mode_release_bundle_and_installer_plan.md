# User-Mode Release Bundle And Installer Plan

## Goal

Turn Blueprint RE from a source-first install flow into a release-first product install flow.

Target operator experience:

```bash
curl -fsSL https://example.com/blueprint-re/install.sh | bash
```

or, for an already downloaded release artifact:

```bash
bash blueprint-re-<version>-linux-x86_64.sh
```

The self-extracting artifact contains an appended binary `tar.gz` payload, so
the product path should download the file first and then execute it. A
`curl | bash` flow should use a separate small text-only downloader stub that
downloads the self-extracting artifact to a temporary file and runs it; the
binary self-extracting artifact itself should not be streamed directly into
Bash.

The installer should set up a runnable local web application for one user without requiring the user to clone the repo, manually create Python environments, manually run `npm install`, or manually wire `systemd --user`.

This plan keeps the current user-mode deployment model:

- install into user-owned directories
- run via `systemd --user`
- keep the executor sandbox requirement as `bwrap`
- avoid mandatory `sudo`
- avoid mandatory `apt`

## Why Change

Current install is still repository-oriented:

- user starts from the source tree
- `scripts/install_blueprint_re.sh` generates repo-local `.env`
- `scripts/deploy_user_systemd.sh` installs Python and Node dependencies from source
- runtime dependency bootstrap is tied to `apt` when host tools are missing

This is acceptable for developers, but it is not how end-user web applications are typically packaged. A more product-like installation path should:

- consume a versioned self-extracting release artifact
- install from a single entrypoint
- isolate runtime tooling inside a dedicated user-owned environment
- guide missing dependencies with clear diagnostics and automatic bootstrap where safe
- avoid host package-manager coupling where possible

## Non-Goals

This document does not propose:

- converting Blueprint RE to a root-installed system service
- shipping `.deb` or `.rpm` in the first step
- replacing `systemd --user`
- removing `nginx`
- removing `bubblewrap`
- changing the default executor away from `pi`

## Product Model

Blueprint RE should be delivered as one self-extracting release artifact.

The public install UX should have two layers:

- a small text-only downloader/bootstrap script that is safe to run through `curl | bash`
- a versioned self-extracting installer artifact that contains the application payload

The downloader is the user-facing convenience entrypoint. The self-extracting
artifact is the authoritative install artifact. The downloader must never append
or stream the binary payload into Bash; it should download the release artifact
to a temporary file, verify it, and execute it as a normal file.

There are only two realistic "single file" strategies for this application:

- self-extracting archive: a shell stub plus an embedded payload archive
- binary freezing: PyInstaller/pkg-style packaging

Because Blueprint RE contains a Python backend, a Next.js frontend, a Node manager-agent, `nginx`, and `bubblewrap`, binary freezing provides poor value and makes debugging harder. The product installer should therefore use the self-extracting archive model.

### Self-Extracting Artifact

The published artifact should be a versioned executable shell file, for example:

```text
blueprint-re-<version>-linux-x86_64.sh
```

Implementation can use `makeself` or a hand-written shell stub that locates an archive marker and extracts the appended `tar.gz` payload.

The payload should contain only what is needed to deploy the application, not the entire git repository history.

Suggested contents:

```text
blueprint-re/
  release.json
  wheels/
    blueprint_re_backend-<version>-py3-none-any.whl
    *.whl                    # full pip dependency wheel closure
  frontend-standalone/
    frontend/
      server.js               # generated by Next.js standalone build
      .next/                  # traced standalone output
      node_modules/           # traced runtime dependencies when emitted
      public/
  manager-agent/
    src/
    package.json
    package-lock.json
    node_modules/             # production dependencies for tagged releases
  deploy/
  scripts/
  runtime/
    environment.yml
    packages/                 # optional offline package cache
```

Payload requirements:

- include the backend as a built wheel plus the full pip dependency wheel set
- include the prebuilt Next.js standalone server generated by `next build`,
  including traced runtime dependencies, `.next/static`, and `public`
- include manager-agent source, lockfile, and production `node_modules` for
  tagged releases
- include deploy templates and service files
- include installer support scripts
- include a machine-readable release manifest with version/build metadata
- include runtime dependency metadata for online and offline environment creation

The shell stub is the authoritative product installer. The embedded payload is
the release bundle. The downloader script is only a convenience entrypoint that
fetches and launches this artifact. This keeps the user-facing model simple
while preserving a debuggable archive internally.

### Downloader Bootstrap Script

Publish a separate pure-text script, for example:

```text
install.sh
```

This script is allowed to be installed with:

```bash
curl -fsSL https://example.com/blueprint-re/install.sh | bash
```

Responsibilities:

- detect supported OS and architecture
- select a release version or channel, such as `stable`, `latest`, or an explicit version
- check for `curl` or `wget`
- check whether the host can reach the release download endpoint
- download `blueprint-re-<version>-linux-x86_64.sh` to a temporary file
- download and verify the matching `.sha256` or manifest signature
- make the downloaded artifact executable
- execute the artifact and forward installer arguments
- clean up the temporary artifact unless the user requests `--keep-installer`

The downloader should not:

- extract payloads
- create Conda environments
- write service files
- call `sudo`
- call `apt-get`
- mutate `~/.local/share/blueprint-re/` or `~/.config/blueprint-re/` directly

This keeps the network bootstrap path simple, inspectable, and recoverable.

## Packaging Strategy

The install path should stop assuming system packages are the primary bootstrap mechanism.

Preferred runtime dependency source:

- `conda-forge` for:
  - `python >= 3.13`
  - `nodejs >= 22.19`
  - `nginx`
  - `bubblewrap`
  - `git`

This does not eliminate all host assumptions. The host still must support:

- Linux
- `systemd --user`
- functional `bubblewrap` runtime behavior on the host kernel

Important constraint:

Even if `bubblewrap` is installed from Conda, host namespace or setuid support can still fail. The installer must keep a real `bwrap` smoke test and fail explicitly if sandboxing cannot work.

### Dependency Guidance Boundaries

The installer can automatically acquire user-space runtime dependencies. It
cannot safely repair every host-level prerequisite without becoming a privileged
system installer.

Automatically handled by the product installer:

- Python runtime inside `~/.local/share/blueprint-re/env`
- Node.js runtime inside `~/.local/share/blueprint-re/env`
- `nginx` binary inside `~/.local/share/blueprint-re/env`
- `bubblewrap` binary inside `~/.local/share/blueprint-re/env`
- backend wheel and vendored Python dependency wheels
- prebuilt Next.js standalone frontend
- manager-agent dependencies when bundled in the release artifact
- user-mode service unit generation

Checked but not silently fixed:

- missing or unusable `systemd --user`
- missing user session bus
- host kernel or security policy blocking `bubblewrap`
- missing network access when the artifact does not include offline caches
- occupied local ports
- unsupported OS or CPU architecture
- missing required application secrets such as `BLUEPRINT_DEEPSEEK_API_KEY`

For these cases the installer should print a targeted diagnostic and next step.
It should not attempt a privileged package-manager repair by default.

Suggested dependency decision matrix:

| Dependency or capability | Checked by | Auto-handled | Default failure behavior |
| --- | --- | --- | --- |
| Linux OS | downloader and installer | no | stop with unsupported OS message |
| CPU architecture | downloader and installer | no | stop and suggest supported artifact |
| writable `HOME` | installer | no | stop and show target directories |
| `curl` or `wget` | downloader | no | print manual download command |
| `tar` | installer | no | stop with required host tool message |
| `sha256sum` | installer | no | stop with required host tool message |
| `systemd --user` | installer | no | stop and explain user-session requirement |
| micromamba/conda manager | installer | yes, if online or embedded | bootstrap micromamba or stop in `--offline` |
| Python runtime | installer | yes | install into dedicated env |
| Node.js runtime | installer | yes | install into dedicated env |
| `nginx` binary | installer | yes | install into dedicated env |
| `bubblewrap` binary | installer | yes | install into dedicated env |
| `bubblewrap` can run | installer | no | stop with sandbox diagnostic |
| local app ports | installer/deploy | no | stop or require explicit alternate ports |
| DeepSeek API key | installer/deploy | no | stop with config instructions |

### Productized Dependency Policy

The release installer should separate dependencies into three groups:

- product-managed runtime dependencies installed under the Blueprint release
  environment
- host capabilities that cannot be packaged and must be checked
- optional integrations that should never block the default product install

Product-managed dependencies should be resolved from the dedicated environment:

```text
~/.local/share/blueprint-re/env/
```

Generated service units should use absolute paths from that environment instead
of relying on shell activation or host `PATH`.

#### Product-Managed Runtime Dependencies

These should be installed or bundled by the product installer.

| Dependency | Product path | Notes |
| --- | --- | --- |
| Python app runtime | dedicated env | Required for FastAPI backend and installer-side Python helpers after bootstrap. |
| Node.js runtime | dedicated env | Required for Next.js standalone and manager-agent. |
| npm | dedicated env | Required only when manager-agent dependencies are not already bundled. Tagged releases should avoid install-time registry access where possible. |
| nginx | dedicated env | Required gateway. Run as a user service with generated config, temp paths, logs, and pid under `~/.config/blueprint-re/`. |
| bubblewrap / `bwrap` | dedicated env | Required executor sandbox binary. Must still pass host namespace/mount smoke tests. |
| git | dedicated env | Useful for runtime tools/executors and avoids depending on host `git`. |
| micromamba | bundled or verified download | Preferred bootstrap manager. Avoid arbitrary host Conda discovery as the primary product path. |
| backend wheel dependencies | release `wheels/` | Install with `pip --no-index --find-links` from the payload. |
| manager-agent dependencies | release payload | Prefer bundled `node_modules` for tagged releases or a future standalone Node bundle. |
| Next.js standalone output | release payload | Prebuilt in CI; end-user install should not compile frontend assets. |

#### Host Capabilities To Check

These are not normal package dependencies and should not be silently repaired by
the product installer.

| Capability | Why it is host-owned | Installer behavior |
| --- | --- | --- |
| Linux kernel | Runtime target | Fail on unsupported OS. |
| CPU architecture | Artifact compatibility | Select matching artifact or fail. |
| writable `HOME` | User-mode install root | Fail with target path diagnostics. |
| `systemd --user` | Service manager/session capability | Fail with clear user-session diagnostic. |
| user DBus/session manager | Required by `systemd --user` | Fail with login/session guidance. |
| user namespace / mount namespace support | Required by `bwrap` | Run smoke test and fail if blocked. |
| localhost port availability | Runtime binding choice | Check user-facing port first; auto-resolve or report internal conflicts. |
| network access | Needed for online bootstrap | Check only when no offline/bundled cache is available. |
| `curl` or `wget` | Downloader bootstrap | If missing, print manual download instructions. |
| basic POSIX shell tools | Archive and checksum handling | Require `tar`, `sha256sum`, `mktemp`, `sed`, and `grep`. |

#### Optional Integrations

These should be detected and wired when present, but they must not block the
default install path.

| Integration | Behavior |
| --- | --- |
| `pi` CLI | Default executor path uses the bundled `blueprint_pi_launch.sh`; host `pi` detection may be used for diagnostics. |
| `opencode` | Optional executor integration only. |
| `claude` / `claude_code` | Optional `cli_native` integration only. |
| `codex` | Manual-only integration; never auto-generate a default command. |
| host Python/R analysis envs | User/project runtime configuration, not app install prerequisites. |
| Tavily/web search key | Optional feature flag, not an install blocker. |

#### Nginx Productization

`nginx` should be treated like `bwrap`: product-managed binary, host capability
checked only where necessary.

Productized nginx layout:

```text
~/.local/share/blueprint-re/env/bin/nginx
~/.config/blueprint-re/nginx.conf
~/.config/blueprint-re/nginx.pid
~/.config/blueprint-re/nginx-access.log
~/.config/blueprint-re/nginx-error.log
~/.config/blueprint-re/nginx-tmp/
```

Requirements:

- never require or modify system nginx
- never call `sudo systemctl restart nginx`
- bind the product gateway to an unprivileged localhost port, such as
  `127.0.0.1:13001`
- write user-owned temp paths so nginx does not depend on `/var/lib/nginx`
- run `nginx -t -c ~/.config/blueprint-re/nginx.conf` before service start
- surface the gateway URL as the only user-facing port

Binding privileged ports such as `80` or `443` is out of scope for the default
user-mode installer. Operators who need those ports can add their own reverse
proxy or privileged service outside the product installer.

### Executor Sandbox Productization

The release installer should keep `bwrap` as the default executor sandbox
implementation. The problem to solve is not replacing `bwrap`; it is making the
`bwrap` path feel like a product dependency instead of a system-administration
task.

`bwrap` is still the best fit for the current executor model because it can:

- run without root
- isolate `HOME`, `XDG_*`, cache, and temp directories per run
- bind project, run, result, script, and `data_mount` paths explicitly
- preserve normal host Python/R/Conda runtime access when configured
- enforce read-only and writable path boundaries

The installer should productize this path as follows:

- install the `bwrap` binary into the dedicated runtime environment through the
  same runtime bootstrap path as Python, Node, and nginx
- resolve generated services to the dedicated environment's `bwrap` path
- run a real sandbox smoke test during install, not only a `command -v` check
- classify failures clearly as binary missing, namespace blocked, mount blocked,
  or container/host policy blocked
- fail explicitly when `bwrap` cannot work
- never silently downgrade release installs to unsandboxed execution

Acceptable escape hatches:

- a clearly named developer-only unsafe mode, such as
  `--dev-unsafe-no-sandbox`
- a future executor backend setting for controlled deployments, such as
  `local-bwrap`, `local-container`, `remote-hpc`, or `dev-unsafe`

The product default remains:

```text
BLUEPRINT_EXECUTOR_SANDBOX_MODE=bwrap
```

Network isolation is intentionally not enabled in the current executor sandbox.
The `pi` executor and optional CLI executors need outbound API access. The
sandbox plan should document this explicitly with `network=host` and
`network_isolation=false` so users understand the tradeoff.

### Offline Mode

"Single file" should support offline installation for the product path.

Tagged self-extracting release artifacts should carry a Conda package cache for the target platform:

```text
runtime/packages/
```

Developer builds may omit this cache to keep iteration fast and artifact size reasonable.

Installer behavior:

- if `runtime/packages/` exists, create the runtime environment from the embedded cache without network access
- if the embedded cache is absent, fall back to online `conda-forge` resolution
- if the installer was invoked with `--offline`, fail rather than using the network

This makes the product artifact genuinely portable within the supported OS/architecture pair. Online mode remains useful for development builds and smaller artifacts.

## Install Layout

Use stable user-owned paths:

```text
~/.local/share/blueprint-re/
  releases/
    <version>/
  current -> releases/<version>
  env/
  data/
  logs/
  snapshots/

~/.config/blueprint-re/
  backend.env
  manager-agent.env
  frontend.env
  nginx.conf
  nginx.pid
  nginx-access.log
  nginx-error.log
  nginx-tmp/

~/.config/systemd/user/
  blueprint-re-backend.service
  blueprint-re-frontend.service
  blueprint-re-manager-agent.service
  blueprint-re-nginx.service
```

Recommended structure:

- release payload lives under `~/.local/share/blueprint-re/releases/<version>/`
- `current` symlink points at the active release
- Conda environment lives at a stable path such as `~/.local/share/blueprint-re/env`
- `data/` stores persistent project state and is not part of any release payload
- `logs/` stores installer, deploy, migration, and diagnostic logs
- `snapshots/` stores pre-upgrade config/data metadata snapshots used for
  rollback and recovery
- user config remains separated from release payload

This enables upgrades without mutating old releases in place.

## Installer Flow

### Phase 0. Downloader Bootstrap

This phase exists only for the public pure-text downloader script. It is skipped
when the user directly runs `blueprint-re-<version>-linux-x86_64.sh`.

Checks:

- shell is compatible enough to run the downloader
- OS is Linux
- architecture maps to a published artifact
- `mktemp`, `chmod`, and either `curl` or `wget` exist
- release endpoint is reachable unless a local artifact path is supplied
- downloaded installer checksum matches the published checksum

Arguments:

- `--version VERSION` selects an exact release
- `--channel stable|latest|dev` selects a release channel
- `--offline` is forwarded to the self-extracting installer and requires a local artifact or an artifact with embedded caches
- `--keep-installer` keeps the downloaded self-extracting file for debugging or reuse
- `--installer-url URL` allows an operator-managed mirror

Failure behavior:

- if no downloader (`curl`/`wget`) exists, print manual download instructions
- if checksum verification fails, delete the artifact and stop
- if `--offline` is used with no local artifact, stop before invoking the installer

### Phase 1. Host Preflight

Check:

- Linux
- `systemd --user`
- architecture support
- writable `HOME`
- required network access if runtime bootstrap needs downloads
- required shell tools: `tar`, `sha256sum`, `mktemp`, `sed`, `grep`
- local port availability for backend, frontend, manager-agent, and nginx proxy ports
- `bwrap` runtime viability after the runtime environment is created

Do not require:

- `sudo`
- `apt`

The preflight should classify checks into:

- hard blockers that cannot be repaired in user mode
- dependencies that can be acquired automatically
- warnings that may be overridden with explicit flags

Examples:

- hard blocker: `systemd --user` is unavailable
- automatic repair: no host Conda exists, but online micromamba bootstrap is allowed
- warning: target HTTP port is occupied, but a custom port env may be supplied

### Phase 2. Runtime Bootstrap

Preferred order:

1. use an embedded micromamba binary from the self-extracting artifact when present
2. use a verified micromamba download when online mode is allowed
3. optionally use an explicitly configured host `micromamba`, `mamba`, or `conda`
   as an operator override
4. create or update the dedicated Blueprint environment

Suggested environment name or path:

```text
~/.local/share/blueprint-re/env
```

### Phase 3. Runtime Package Install

Install into the dedicated environment:

- `python>=3.13`
- `nodejs>=22.19`
- `nginx`
- `bubblewrap`
- `git`

The installer should use absolute paths from that environment after install, not rely on shell activation.

When offline cache exists, the install should use the embedded package directory and explicit package spec. When online mode is used, it should resolve against `conda-forge` from the lock/spec metadata in the payload.

Auto-bootstrap rules:

- if an embedded runtime package cache exists, prefer it
- if `--offline` is set and no cache exists, fail before attempting network access
- if no Conda-like runtime manager exists and online mode is allowed, download micromamba into a temporary directory or user cache
- once the dedicated env exists, resolve all service binary paths from that env
- never rely on shell activation in generated service units

### Phase 4. Payload Validation

The self-extracting shell stub owns extraction. After the stub expands the embedded payload, the installer should validate the expanded payload before modifying the installed release tree.

Validation before deploy:

- confirm `release.json`
- confirm expected architecture/platform
- verify checksums for wheel, frontend, manager-agent, and runtime package cache if present

### Phase 5. Release Deployment

Copy or move the already-expanded payload to:

```text
~/.local/share/blueprint-re/releases/<version>/
```

Then atomically repoint:

```text
~/.local/share/blueprint-re/current
```

### Phase 6. Config Generation

Generate:

- `backend.env`
- `manager-agent.env`
- `frontend.env`
- `nginx.conf`

Runtime env generation should remain whitelist-based for backend service safety.

### Phase 7. Service Install And Start

Generate user service units with absolute paths:

- Python from the dedicated env
- Node/NPM from the dedicated env
- `nginx` from the dedicated env
- app root from `~/.local/share/blueprint-re/current`

Then:

- `systemctl --user daemon-reload`
- `enable` services
- start or restart manager-agent, backend, frontend, then nginx
- start nginx last after config validation

Initial install may start services directly. Upgrade should use the stop-the-world flow described in the upgrade section.

### Phase 8. Verification

At minimum:

```bash
systemctl --user status blueprint-re-backend.service --no-pager
systemctl --user status blueprint-re-manager-agent.service --no-pager
systemctl --user status blueprint-re-frontend.service --no-pager
systemctl --user status blueprint-re-nginx.service --no-pager
curl -fsS http://127.0.0.1:18001/healthz
curl -I http://127.0.0.1:13001
```

## Required Refactor Of Existing Scripts

Current scripts combine too many concerns:

- interactive config collection
- dependency discovery
- host package installation
- source build
- service generation
- service restart

These concerns should be split.

### Proposed Script Roles

#### `scripts/install_downloader.sh`

Public text-only network bootstrap entrypoint.

Responsibilities:

- implement `curl | bash` friendly install UX
- resolve release version/channel to a concrete artifact
- download the self-extracting installer
- verify checksum before execution
- forward installer arguments
- provide manual fallback instructions when download tools are missing

This script must not contain an appended binary payload.

#### `install.sh`

Public product entrypoint inside the self-extracting artifact.

Responsibilities:

- extract embedded payload
- validate installer version against payload version
- bootstrap runtime manager from embedded or host micromamba/conda
- install Conda packages
- validate expanded payload
- install backend wheel from local wheels using `--no-index --find-links`
- call deploy logic

#### `scripts/deploy_release.sh`

Release deploy entrypoint.

Responsibilities:

- consume an already prepared runtime environment
- generate env files
- install service units
- build or install release-local app payload if needed
- restart services

This script should not call `apt-get`.

#### `scripts/render_runtime_config.sh`

Optional helper to isolate env file rendering and config templating.

#### `scripts/build_release_bundle.sh`

Release authoring tool.

Responsibilities:

- build backend wheel and full pip dependency wheel set
- build prebuilt frontend standalone output and copy traced runtime output,
  `.next/static`, and `public`
- gather manager-agent payload with production `node_modules` for tagged
  releases
- gather runtime metadata and optional offline package cache
- emit payload tarball
- write `release.json`

#### `scripts/build_self_extracting_installer.sh`

Release authoring tool.

Responsibilities:

- call or consume `scripts/build_release_bundle.sh`
- prepend installer shell stub
- append payload archive
- write final `blueprint-re-<version>-linux-x86_64.sh`

## Changes Needed In Current Deploy Logic

The current deploy script should be changed in these ways.

### 1. Remove Mandatory `apt` Bootstrap From Deploy Path

Current behavior:

- `scripts/deploy_user_systemd.sh` tries `apt-get install ... nginx`

Target behavior:

- deploy script only validates dependencies
- installer script is responsible for acquiring them
- optional fallback `--allow-apt` can remain as an operator-only escape hatch
- product downloader and product self-extracting installer do not pass `--allow-apt`

### 2. Accept Explicit Binary Paths

Current binary lookup is host-PATH-centric.

Add explicit support for variables such as:

- `BLUEPRINT_PYTHON_BIN`
- `BLUEPRINT_NODE_BIN`
- `BLUEPRINT_NPM_BIN`
- `BLUEPRINT_NGINX_BIN`
- `BLUEPRINT_BWRAP_BIN`

If set, deploy should trust them before `command -v`.

This is important for Conda-based installs because services should use stable absolute paths from the dedicated environment rather than relying on shell activation.

### 3. Stop Requiring Repo-Local Install State

The deploy path should not assume the application lives in a git checkout that users keep around permanently.

Instead it should operate on a release root, for example:

```text
BLUEPRINT_RELEASE_ROOT=~/.local/share/blueprint-re/current
```

### 4. Require Prebuilt Frontend Artifacts

For a product install, shipping prebuilt frontend output is required. End-user installs should not compile Next.js.

Target behavior:

- tagged release artifacts must include prebuilt frontend standalone output
- local source builds may keep a developer-only fallback path
- the product installer fails if the prebuilt frontend artifact is missing

This reduces install time, avoids unnecessary local toolchain issues, and moves more failure surface into CI where it belongs.

### 5. Require Backend Wheel

The backend should ship as a built wheel in product releases.

Target behavior:

- release build produces `wheels/blueprint_re_backend-<version>-py3-none-any.whl`
- installer installs the wheel into the dedicated runtime environment
- product installer should not use `pip install -e`
- editable install remains a developer-only source checkout path

This reduces install-time variability and avoids coupling user installs to source tree layout.

### 6. Keep User Install Simple

The product install command should remain:

```bash
curl -fsSL https://example.com/blueprint-re/install.sh | bash
```

The implementation can remain multi-stage internally, but the user should not
need to know about Conda, wheel installation, Next.js standalone output, nginx
templates, or systemd unit wiring.

Installer output should be structured around user decisions:

- what will be installed
- where it will be installed
- which dependency checks passed
- which dependency checks were automatically handled
- which host-level blockers require manual action
- how to open the app after install
- how to uninstall or rollback

Avoid exposing low-level logs by default. Keep detailed logs under:

```text
~/.local/share/blueprint-re/logs/
```

## Upgrade Model

The installer should support idempotent upgrades:

```bash
bash blueprint-re-0.5.1-linux-x86_64.sh
```

Upgrade behavior:

- extract and validate the embedded payload
- unpack into a new release directory
- snapshot persistent state before migration
- run release-defined migration checks
- regenerate env files if needed
- switch `current` symlink
- restart services
- keep at least one previous release for rollback

Rollback option:

```bash
bash blueprint-re-<version>-linux-x86_64.sh --rollback <previous-version>
```

### Persistent State

Release payloads must not contain user state.

Current application state is file-backed under `Settings.data_root`, whose default is the repository `workspace/` in source mode. Product install should set a stable user-mode data root, for example:

```text
BLUEPRINT_DATA_ROOT=~/.local/share/blueprint-re/data
```

The data root contains project registry and project state such as:

- `_system/project_registry.json`
- project directories
- `project.json`
- `graph/`
- `runs/`
- `results/`
- `reports/`
- `chat/`
- `artifacts/`
- `artifact_store/`

The release directory and runtime environment are replaceable. The data root is persistent and must be preserved across upgrades and rollbacks.

### Migration Contract

Each release should declare migration metadata in `release.json`:

```json
{
  "version": "0.5.1",
  "state_schema_version": "0.5.1",
  "migrations": {
    "preflight": "scripts/migrate_preflight.sh",
    "apply": "scripts/migrate_apply.sh",
    "rollback_supported": true
  }
}
```

`version` identifies the release artifact. `state_schema_version` identifies the persistent project-state schema expected by that release. They may be equal for most releases, but they are intentionally separate so patch releases that do not change persisted state can share the same schema version, and schema-only migration releases can be reasoned about explicitly.

Upgrade sequence:

1. stop `blueprint-re-nginx.service` so new browser requests are rejected quickly
2. stop frontend, manager-agent, and backend
3. snapshot `~/.config/blueprint-re/` and the Blueprint data root metadata
4. run migration preflight against the persistent data root
5. apply migrations if preflight passes
6. switch `current` symlink
7. regenerate service units and env files
8. start manager-agent, backend, frontend, then nginx
9. run health checks

If migration or health checks fail:

- stop services
- restore the previous `current` symlink
- restore config snapshot
- restore data snapshot only if the migration declared full rollback support
- start the previous release

Migrations that mutate project state must be idempotent and version-stamped. File-backed migrations should prefer explicit marker files or schema metadata rather than inferring completion from partial file shapes.

### Service Interruption Model

The first production installer should use a short stop-the-world upgrade.

Rolling upgrade is not a good default for the current architecture because:

- frontend, backend, manager-agent, and nginx share local ports and generated env files
- active executor runs can mutate project state
- the backend currently owns file-backed project state under one data root

Upgrade preflight should block or warn when active runs/background jobs exist. The default upgrade should require the system to be idle before mutating persistent state.

### `current` Symlink And Live Requests

The `current` symlink should only change while services are stopped. Running services must not continue serving from a release directory whose symlink has moved.

Expected behavior:

- stop nginx first to prevent new browser/API requests
- stop frontend/backend/manager-agent before switching `current`
- switch `current` atomically with `ln -sfn`
- start services from the new resolved release root
- health check before declaring success

Existing in-flight requests are interrupted during upgrade. That is acceptable for the first product installer and should be documented in the installer output.

## Test Strategy

The installer should be tested as a product artifact, not only as shell script text.

### Script Unit Tests

Use shell-level tests for helper functions:

- argument parsing
- downloader channel/version resolution
- downloader checksum verification
- archive marker detection
- release manifest validation
- explicit binary path resolution
- offline/online mode selection
- rollback target selection
- dependency classification into blocker/auto-bootstrap/warning

These tests should run without touching the real user home.

### Temp Home Integration Tests

Run installer and deploy behavior against a temporary `HOME`.

Required checks:

- generated `~/.config/blueprint-re/*.env`
- generated `~/.config/systemd/user/*.service`
- release unpack layout under `~/.local/share/blueprint-re/releases/<version>/`
- `current` symlink target
- data root stays outside the release directory
- no `apt-get` or `sudo` path is used in the default product install
- downloader executes the downloaded artifact rather than streaming binary payload through Bash

Where `systemd --user` is unavailable in CI, test rendering and validation separately from actual service start.

### Container Tests

Use a Linux container target such as Podman for end-to-end install tests.

Suggested matrix:

- online install with no preexisting Conda
- offline install with embedded package cache
- downloader install from a local HTTP server
- upgrade from previous release
- rollback after failed migration
- host with missing `sudo`
- host with missing `apt-get`
- host with unavailable `systemd --user`
- host with `bwrap` blocked by namespace policy

The container test should verify actual command behavior, not just generated files, when the container can provide `systemd --user` and `bwrap` support. If namespace support is unavailable, the test should assert the installer fails with the expected `bwrap` diagnostic.

### Release Artifact Tests

Every tagged self-extracting artifact should be validated before publication:

- downloader script is pure text and shell syntax checks pass
- downloader can fetch and verify the artifact from the release endpoint
- installer file is executable
- payload marker can be located
- payload checksums match `release.json`
- backend wheel installs into a clean runtime env using only payload-local wheels
- frontend service can start with `node frontend-standalone/frontend/server.js`
- manager-agent passes `node --check src/server.js`
- `nginx -t -c <generated nginx.conf>` passes
- health checks pass after install

### Implementation Gap Audit

The current implementation should be audited against this checklist before a
tagged installer release. Some items are already partially implemented, but they
remain release-contract requirements until covered by artifact tests.

#### P0. Backend Wheel Selection And Dependency Closure

Release payloads must include the backend wheel and the full pip dependency
wheel set.

Required behavior:

- build the backend wheel
- build or download all transitive Python dependency wheels into `wheels/`
- install with `pip install --no-index --find-links <payload>/wheels`
- select the backend wheel by `release.json.artifacts.backend_wheel.path`, not
  by `ls wheels/*.whl | head -n1`
- fail the build if any dependency is only available as an sdist that would
  require install-time compilation

Reason:

`wheels/` contains multiple `.whl` files once dependencies are bundled. The
installer must not infer the application wheel from filesystem sort order.

Preferred release build implementation:

```bash
python3 -m pip wheel --wheel-dir wheels/ backend/
```

The release manifest should record the exact backend wheel path and checksum.
The installer should read that path from the manifest before validating and
installing the wheel.

#### P0. Frontend Standalone Completeness

The Next.js standalone payload must be self-contained.

Required behavior:

- copy `.next/standalone/` into `frontend-standalone/`
- copy `.next/static/` into `frontend-standalone/frontend/.next/static/`
- copy `public/` into `frontend-standalone/frontend/public/`
- verify `frontend-standalone/frontend/server.js` exists
- verify `node frontend-standalone/frontend/server.js` can start in a clean
  release directory

Reason:

Next.js standalone output traces runtime dependencies. With a monorepo
`outputFileTracingRoot`, traced files can live outside the frontend directory.
The release artifact must include the traced output exactly as produced by
`.next/standalone`, plus static and public assets.

#### P0. Manager-Agent Dependency Closure

Tagged releases should not require `npm ci` during end-user install.

Required behavior:

- run `npm ci --omit=dev` for `manager-agent` during release build
- include `manager-agent/node_modules/` in tagged payloads
- make `deploy_release.sh` skip npm install when bundled `node_modules/` exists
- fail offline install if `node_modules/` is absent

Reason:

The manager-agent has a small dependency set and should be made deterministic at
release-build time. Installing it from npm during user install breaks offline
mode and adds registry/network variability.

#### P1. User Service Linger

The installer must decide how persistent the local web app should be after the
user logs out.

Required behavior:

- detect current linger status with `loginctl show-user "$USER" -p Linger`
- if linger is disabled, explain that `systemd --user` services may stop after
  logout
- attempt `loginctl enable-linger "$USER"` only when policy allows it and the
  installer can do so without escalating privileges
- otherwise print a precise manual command or document that services are
  session-scoped on that host

Important constraint:

`loginctl enable-linger` is host-policy dependent. Some distributions require
privileges or administrator policy. The installer should not make linger a hard
install blocker unless the product explicitly requires services to persist after
logout.

#### P1. Downloader And Version Resolution

The one-line install path is not complete until the downloader protocol exists.

Required behavior:

- provide a pure-text `install_downloader.sh`
- publish a release metadata endpoint such as `latest.json`
- define channel-to-version resolution for `stable`, `latest`, and optional
  `dev`
- define artifact URL and checksum URL templates
- verify the downloaded self-extracting installer before execution
- forward installer arguments such as `--offline`, `--rollback`, and
  `--keep-installer`

The binary self-extracting installer must continue rejecting direct pipe input.

#### P1. Source-Mode Data Migration

Existing source-checkout users may have project state under a repository
`workspace/` directory.

Required behavior:

- detect likely source-mode state roots, especially `<repo>/workspace/`
- detect `_system/project_registry.json` and project directories
- offer a migration path into `~/.local/share/blueprint-re/data`
- avoid overwriting existing release-mode data unless the user explicitly
  chooses merge/replace behavior
- document a manual migration fallback

This is not needed for brand-new users, but it matters for converting existing
developer/source installs to release installs.

#### P1. Active Run Upgrade Preflight

Stop-the-world upgrades must not blindly interrupt active executor work.

Required behavior:

- before stopping services, check for active runs/background jobs
- block upgrade by default when active work exists
- provide an explicit override for administrators or emergency rollback
- print active project/run/job identifiers when blocking

Possible implementation paths:

- call a backend health/admin endpoint before shutdown
- inspect project graph state under the data root
- inspect background task state files

The first version can be conservative: if active state cannot be determined, warn
or require an explicit `--force-upgrade`.

#### P1. bwrap Host Compatibility Diagnostics

Installing the `bwrap` binary is not enough. The host must also allow the needed
namespace and mount behavior.

Required behavior:

- run a realistic `bwrap` smoke test after runtime env creation
- keep the release default as `BLUEPRINT_EXECUTOR_SANDBOX_MODE=bwrap`
- classify common failures where possible:
  - binary missing
  - unprivileged user namespace disabled
  - mount namespace blocked
  - container policy/seccomp blocked
  - setuid/userns configuration mismatch
- include the diagnostic in installer output and logs

Do not silently fall back to unsandboxed execution.

#### P2. Systemd Unit Hygiene

The release service units should encode service relationships and safe defaults.

Required behavior:

- add `Requires=` or an equivalent dependency policy where service startup truly
  depends on another service
- add `Conflicts=` or clear port-preflight checks to avoid source-mode and
  release-mode deployments fighting over ports
- add `TimeoutStopSec=10`, `KillMode=control-group`, and `SendSIGKILL=yes`
  consistently where appropriate
- add compatible hardening directives such as `NoNewPrivileges=yes`
- ensure nginx temp directories are recreated before nginx starts, not only at
  deploy time

These changes should be behavior-tested because overly aggressive systemd
hardening can break user-mode nginx, Next.js, or executor access to configured
runtime paths.

#### P2. Same-Version Reinstall Semantics

Reinstalling the same version should be explicit.

Required behavior:

- detect same-version reinstall
- decide whether it is a repair install, replace install, or no-op
- preserve rollback/snapshot behavior when replacing an existing release
- avoid treating backup directories as release versions

The installer should not silently skip migration/snapshot protections when
overwriting the active release.

## Uninstall Model

Provide a dedicated uninstall command, for example:

```bash
bash uninstall.sh
```

Responsibilities:

- stop and disable user services
- remove user service unit files
- remove release/runtime files under `~/.local/share/blueprint-re/`
- preserve `~/.local/share/blueprint-re/data/` by default, or require an
  explicit destructive flag such as `--purge-data`
- warn clearly before deleting persistent project state
- optionally preserve `~/.config/blueprint-re/` unless `--purge-config`

## Security And Runtime Constraints

The installer must preserve these repository rules:

- `BLUEPRINT_EXECUTOR_SANDBOX_MODE=bwrap` remains required
- no silent fallback to unsandboxed execution
- secrets stay out of logs
- managed deploy still requires `BLUEPRINT_DEEPSEEK_API_KEY`
- `codex` remains manual-only in deploy defaults

### Security Hardening Backlog

The release installer should treat the following security items as part of
productization, not as optional polish.

| Priority | Item | Required action |
| --- | --- | --- |
| P0 | Generated env files contain secrets | Write `backend.env`, `manager-agent.env`, and other service env files with restrictive permissions; use `umask 077` and/or `chmod 600` after generation. |
| P0 | Online micromamba bootstrap integrity | Do not use unauthenticated `curl | tar` for micromamba. Prefer a bundled micromamba binary covered by payload checksums, or download a pinned artifact and verify SHA-256/signature before execution. |
| P1 | Uninstall data loss risk | `uninstall.sh` must clearly warn that `~/.local/share/blueprint-re/data/` contains persistent project state and should preserve data by default or require an explicit destructive flag. |
| P1 | Basic systemd service hardening | Add safe hardening directives where compatible, starting with `NoNewPrivileges=yes` and private temp/runtime isolation that does not break nginx, frontend, backend, or manager-agent. |
| P1 | Migration hooks execute code | Migration hooks are release code and are protected by artifact checksums, but they still run with user privileges. Prefer a constrained migration contract; until then, log execution, pass explicit roots, and consider running hooks through the same `bwrap` productized sandbox. |
| P2 | Legacy deploy secret rendering | Developer/source deploy scripts should be marked developer-only and gradually moved to the same env escaping and permission model as `deploy_release.sh`. |
| P2 | Executor network access | Current `bwrap` sandbox uses host networking by design. Keep this explicit in `sandbox_plan.json` and revisit network policy only after executor API access and provider routing are abstracted. |

Minimum v1 release blocker list:

- generated env files are not world-readable
- downloaded or bundled micromamba is integrity-checked
- release install never silently disables `bwrap`
- uninstall does not silently destroy persistent project state

The following simplifications are acceptable after the release path is proven:

- defer full offline Conda package cache support to a tagged-release/CI path
- defer a full migration framework until the first real state migration exists
- simplify runtime bootstrap to bundled micromamba or verified micromamba
  download, instead of relying on arbitrary host Conda discovery
- mark legacy source checkout deploy scripts as developer-only once the release
  installer is stable

## Compatibility Notes

This user-mode installer is the right next step for productization, but it is still not the same as a classic root-installed server package.

Compared with tools such as `rstudio-server`:

- this plan is user-scoped, not system-scoped
- this plan avoids root installation
- this plan uses `systemd --user`, not system services
- this plan keeps all runtime state under the user account

That tradeoff is intentional because it better matches the current architecture and avoids turning install into a privileged operations task.

## Recommended Implementation Sequence

1. Refactor existing deploy script so it can consume explicit runtime binary paths and no longer owns `apt` installation.
2. Add backend wheel build support for release packaging.
3. Make backend dependency closure deterministic by packaging the full pip wheel set and selecting the backend wheel from `release.json`.
4. Move frontend build to release-time and require self-contained standalone output in product artifacts.
5. Build manager-agent dependencies at release time and bundle production `node_modules`.
6. Add release payload build tooling with `release.json`, runtime specs, and tagged-release offline package cache.
7. Add self-extracting installer build tooling.
8. Teach deploy to work from a release root instead of a live git checkout.
9. Add dependency classification and clear diagnostics for host-level blockers, including `bwrap` compatibility diagnostics.
10. Add user service linger detection and installer messaging.
11. Add the pure-text downloader/bootstrap script and release metadata protocol for the public one-line install UX.
12. Add source-mode data migration detection and migration option.
13. Add active-run/background-job upgrade preflight.
14. Add migration preflight/apply hooks and data-root snapshot handling.
15. Add systemd unit hardening and service relationship cleanup.
16. Add tmp `HOME` and container-level installer tests.
17. Add upgrade, rollback, same-version reinstall, and uninstall commands.
